  DATA FLOW FOR A SINGLE CLAIM VERIFICATION:
  ─────────────────────────────────────────
    1. User types claim in React → clicks "Verify"
    2. React POSTs to → localhost:5000/api/claims/verify
    3. Backend auth middleware checks JWT
    4. Backend hashes claim text (md5) → checks Redis  claim_result:<hash>
    5. If Redis hit → return cached verdict instantly (skip steps 6-10)
    6. INSERT into MySQL  claims  table (status: 'pending')
    7. Backend POSTs to → localhost:5001/verify (NLP service)
    8. NLP runs Tier 1 → maybe Tier 2 → maybe Tier 3 → returns result
    9. INSERT into MySQL  verification_log  (full audit trail)
   10. UPDATE MySQL  claims  SET verdict, confidence, status='verified'
   11. SET Redis  claim_result:<hash>  = result (TTL 24 hours)
   12. Return JSON response to React frontend

  STEP 6 — backend/middleware/auth.js (JWT middleware)
  ─────────────────────────────────────────────
    What: Middleware that runs on protected routes. Extracts JWT from
          Authorization header, verifies it, attaches user info to req.user.

    Flow:
      1. Read header: Authorization: Bearer <token>
      2. jwt.verify(token, process.env.JWT_SECRET)
      3. Check Redis: GET jwt_blacklist:<jti>
         → if key exists, token was logged out → return 401
      4. Attach decoded payload to req.user = { id, email, role }
      5. next()

    Error cases:
      - No token → 401 "No token provided"
      - Invalid/expired → 401 "Invalid token"
      - Blacklisted → 401 "Token has been revoked"

  STEP 7 — backend/controllers/authController.js
  ─────────────────────────────────────────────
    Three functions:

    register(req, res):
      1. Read { name, email, password } from req.body
      2. Check if email already exists: SELECT * FROM users WHERE email = ?
      3. Hash password: bcrypt.hash(password, 10)
      4. INSERT INTO users (name, email, password_hash) VALUES (?, ?, ?)
      5. Generate JWT: jwt.sign({ id, email, role }, JWT_SECRET, { expiresIn: '7d' })
         Include a unique jti (JWT ID) for logout support:
         const jti = crypto.randomUUID();
         jwt.sign({ id, email, role, jti }, ...)
      6. Return { token, user: { id, name, email, role } }

    login(req, res):
      1. Read { email, password } from req.body
      2. SELECT * FROM users WHERE email = ?
      3. bcrypt.compare(password, user.password_hash)
      4. Generate JWT (same as register, include jti)
      5. SET Redis session:<userId> with TTL 7d
      6. Return { token, user: { id, name, email, role } }

    logout(req, res):
      1. Read token from Authorization header
      2. Decode to get jti and exp
      3. Calculate remaining TTL: exp - now (in seconds)
      4. SET Redis jwt_blacklist:<jti> "revoked" EX <remaining_ttl>
      5. DEL Redis session:<userId>
      6. Return { message: "Logged out" }

  STEP 9 — backend/controllers/claimController.js
  ─────────────────────────────────────────────
    This is the MOST IMPORTANT file. Core function: submitClaim.

    submitClaim(req, res):
      1. Read { text } from req.body
      2. userId = req.user.id (from JWT middleware)
      3. Hash the claim: crypto.createHash('md5').update(text).digest('hex')
      4. Check Redis dedup:
           const cached = await redis.get(`claim_result:${hash}`);
           if (cached) return res.json(JSON.parse(cached));
      5. INSERT INTO claims (user_id, original_text, status) VALUES (?, ?, 'pending')
         Get insertId → claimId
      6. Call NLP service:
           const nlpResult = await axios.post(
             `${process.env.NLP_SERVICE_URL}/verify`,
             { text }
           );
      7. INSERT INTO verification_log:
           (claim_id, official_value, claimed_value, difference,
            percentage_error, verdict, tier_used, confidence,
            evidence_json, explanation)
      8. UPDATE claims SET
           extracted_metric = ?, extracted_value = ?,
           extracted_year = ?, credibility_score = ?,
           verdict = ?, status = 'verified'
         WHERE id = claimId
      9. Cache in Redis:
           await redis.set(`claim_result:${hash}`, JSON.stringify(response), 'EX', 86400);
     10. Return full result to frontend

    submitQuick(req, res):
      Same flow but call /verify/quick instead of /verify.

    submitDeep(req, res):
      Same flow but call /verify/deep instead of /verify.

    getUserClaims(req, res):
      SELECT * FROM claims WHERE user_id = ? ORDER BY created_at DESC

    getClaimById(req, res):
      SELECT c.*, v.* FROM claims c
      LEFT JOIN verification_log v ON v.claim_id = c.id
      WHERE c.id = ? AND c.user_id = ?

    getStats(req, res):
      1. Check Redis: GET stats_cache → if hit, return it
      2. Run aggregate queries:
         SELECT verdict, COUNT(*) as count FROM claims
         WHERE user_id = ? GROUP BY verdict
         SELECT AVG(credibility_score) FROM claims WHERE user_id = ?
      3. Cache in Redis: SET stats_cache EX 600 (10 min)
      4. Return { total, accurate, misleading, false, avg_confidence }

SECTION 4 — DATABASE SCHEMA (all MySQL tables)
================================================================================

  EXISTING TABLES (already in schema.sql):
  ─────────────────────────────────────────

  users
    id              INT AUTO_INCREMENT PRIMARY KEY
    name            VARCHAR(100) NOT NULL
    email           VARCHAR(150) UNIQUE NOT NULL
    password_hash   VARCHAR(255) NOT NULL
    role            ENUM('user','admin') DEFAULT 'user'
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP

  claims
    id              INT AUTO_INCREMENT PRIMARY KEY
    user_id         INT NOT NULL → FK users(id)
    original_text   TEXT NOT NULL
    extracted_metric VARCHAR(200)
    extracted_value DECIMAL(15,2)
    extracted_year  INT
    credibility_score DECIMAL(5,2)
    status          ENUM('pending','verified','failed') DEFAULT 'pending'
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP

  official_data_cache
    id              INT AUTO_INCREMENT PRIMARY KEY
    metric_name     VARCHAR(200) NOT NULL
    year            INT NOT NULL
    value           DECIMAL(15,2) NOT NULL
    source          VARCHAR(300)
    last_updated    TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE
    UNIQUE KEY (metric_name, year)

  verification_log
    id              INT AUTO_INCREMENT PRIMARY KEY
    claim_id        INT NOT NULL → FK claims(id)
    official_value  DECIMAL(15,2)
    claimed_value   DECIMAL(15,2)
    difference      DECIMAL(15,2)
    percentage_error DECIMAL(5,2)
    verdict         ENUM('accurate','misleading','false')
    verified_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP

  ─────────────────────────────────────────

  ─────────────────────────────────────────
  NEW TABLES TO CREATE:
  ─────────────────────────────────────────

  CREATE TABLE trending_stories (
    id              INT AUTO_INCREMENT PRIMARY KEY,
    headline        TEXT NOT NULL,
    claim_text      TEXT,
    source_name     VARCHAR(255),
    source_url      TEXT,
    published_at    DATETIME,
    fetched_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    verdict         ENUM('accurate','misleading','false','unverifiable'),
    confidence      FLOAT,
    danger_score    FLOAT DEFAULT 0,
    metric          VARCHAR(100),
    official_value  FLOAT,
    claimed_value   FLOAT,
    pct_error       FLOAT,
    explanation     TEXT,
    evidence_json   JSON,
    tier_used       VARCHAR(20),
    is_active       TINYINT(1) DEFAULT 1,
    INDEX idx_danger (danger_score DESC),
    INDEX idx_fetched (fetched_at DESC),
    INDEX idx_active (is_active)
  );

  CREATE TABLE source_stats (
    id               INT AUTO_INCREMENT PRIMARY KEY,
    source_name      VARCHAR(255) UNIQUE NOT NULL,
    total_claims     INT DEFAULT 0,
    accurate_count   INT DEFAULT 0,
    misleading_count INT DEFAULT 0,
    false_count      INT DEFAULT 0,
    avg_danger_score FLOAT DEFAULT 0,
    last_updated     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
  );

SECTION 5 — REDIS USAGE MAP
================================================================================

  KEY PATTERN              │ TTL    │ SET BY             │ READ BY            │ PURPOSE
  ─────────────────────────┼────────┼────────────────────┼────────────────────┼─────────────────────────
  claim_result:<md5hash>   │ 24h    │ claimController    │ claimController    │ Dedup: same claim within
                           │        │                    │                    │ 24h → skip NLP call
  ─────────────────────────┼────────┼────────────────────┼────────────────────┼─────────────────────────
  trending_feed            │ 5min   │ trendingController │ trendingController │ Cache trending page
                           │        │ trendingJob        │                    │ response (expensive query)
  ─────────────────────────┼────────┼────────────────────┼────────────────────┼─────────────────────────
  rate_limit:<ip>          │ 15min  │ express-rate-limit │ express-rate-limit │ 100 req/15min per IP
  ─────────────────────────┼────────┼────────────────────┼────────────────────┼─────────────────────────
  jwt_blacklist:<jti>      │ = exp  │ authController     │ auth middleware    │ Logout: revoke a JWT
                           │        │ (logout)           │                    │ before its natural expiry
  ─────────────────────────┼────────┼────────────────────┼────────────────────┼─────────────────────────
  session:<userId>         │ 7d     │ authController     │ auth middleware    │ Track active sessions
                           │        │ (login)            │                    │
  ─────────────────────────┼────────┼────────────────────┼────────────────────┼─────────────────────────
  stats_cache              │ 10min  │ claimController    │ claimController    │ Dashboard aggregate stats
  ─────────────────────────┼────────┼────────────────────┼────────────────────┼─────────────────────────
  trending_job_lock        │ 5min   │ trendingJob        │ trendingJob        │ Prevent overlapping cron
                           │        │                    │                    │ runs (SET NX)

  DECISION RULE — WHEN TO USE REDIS vs MYSQL:
    Need SQL joins or complex queries?        → MySQL
    Need < 1ms lookup with simple key?        → Redis
    Expensive computed result, cacheable?      → Redis cache with TTL
    Data must survive server restart?          → MySQL (Redis is ephemeral)
    Always write to MySQL FIRST, then cache in Redis.

SECTION 8 — TRENDING RUMOURS FEATURE (full spec)
================================================================================

  PURPOSE:
    Continuously monitor news feeds and fact-check databases, auto-run the
    RAV engine on every story, and surface the most DANGEROUS / MISLEADING
    stories right now — ranked by harm potential.

  ── ARCHITECTURE FLOW ────────────────────────────────────────────────────

    NewsAPI /everything  ─┐
                          ├─→  Node.js cron job (every 30 min)
    Google Fact Check  ───┘         │
                                    │  POST each headline to HF model:
                                    │  https://eyesee11-b-ware.hf.space/verify
                                    ▼
                            Store verdict in MongoDB/MySQL
                            trending_stories table
                                    │
                                    ▼
                            GET /api/trending  (Node backend)
                                    │
                                    ▼
                            React /trending page
                            Card grid: headline + verdict badge + danger bar

  ── DATA SOURCES (EXACT API CALLS) ───────────────────────────────────────

  SOURCE 1 — NewsAPI /everything  (best coverage on free tier)
  ─────────────────────────────────────────────────────────────
  ⚠️  Use /everything NOT /top-headlines — /top-headlines returns 0 results
      for country=in on the free developer plan.

  Rotate through these queries (run one per cron tick to spread load):

    GET https://newsapi.org/v2/everything
      ?q=India+GDP+growth+2025
      &language=en
      &sortBy=publishedAt
      &pageSize=20
      &apiKey=5f146fc58909409caeffba7c49a476fe

    GET https://newsapi.org/v2/everything
      ?q=India+unemployment+rate
      &language=en&sortBy=publishedAt&pageSize=20
      &apiKey=5f146fc58909409caeffba7c49a476fe

    GET https://newsapi.org/v2/everything
      ?q=India+inflation+RBI
      &language=en&sortBy=publishedAt&pageSize=20
      &apiKey=5f146fc58909409caeffba7c49a476fe

    GET https://newsapi.org/v2/everything
      ?q=India+poverty+rate+OR+India+fiscal+deficit
      &language=en&sortBy=publishedAt&pageSize=20
      &apiKey=5f146fc58909409caeffba7c49a476fe

  From each article use:  title + ". " + description  as the claim text.
  Skip articles where description is null.

  SOURCE 2 — Google Fact Check API  (BETTER for viral/social-media claims)
  ─────────────────────────────────────────────────────────────────────────
  This indexes claims already fact-checked by AFP, AltNews, BOOM, Snopes.
  These are claims that went viral enough to be investigated — ideal source.

    GET https://factchecktools.googleapis.com/v1alpha1/claims:search
      ?query=India+economy
      &languageCode=en
      &pageSize=20
      &key=AIzaSyAQOu_R1kR_m6QBMhM6p2zN6XNjTmYYXx0

  From each result:
    claim text     → claimReview[0].claim.text       → send to /verify
    existing label → claimReview[0].textualRating     → "False"/"Misleading"
    source         → claimReview[0].publisher.name
    url            → claimReview[0].url

  ── DANGER SCORE FORMULA (0–100) ─────────────────────────────────────────

    verdict_weight   : accurate=0, misleading=40, false=80, unverifiable=15
    confidence_bonus : confidence × 20   (max 20 points)
    recency_bonus    : published < 2h ago → +10, < 24h → +5, else 0
    spread_bonus     : (source_count - 1) × 5

    Example: false claim (80) + 0.9 conf (+18) + 3 sources (+10) + 1h ago (+10)
             = 118 → capped at 100

  ── BACKEND CRON FLOW (every 30 min) ─────────────────────────────────────

    1. GET NewsAPI /everything → 20 latest articles
    2. GET Google Fact Check   → 20 latest claims
    3. Deduplicate by URL (skip if already in trending_stories)
    4. For each new article:
         a. Build claim text: article.title + ". " + article.description
         b. POST https://eyesee11-b-ware.hf.space/verify  { text: claimText }
         c. Filter: only save if verdict === 'false' || 'misleading'
         d. Compute danger_score
         e. UPSERT into trending_stories (findOneAndUpdate with upsert:true)
    5. DEL Redis trending_feed (invalidate cache)
    6. UPDATE trending_stories SET is_active=0 WHERE fetched_at < NOW() - 48h

  KEY PACKAGES:
    npm install node-cron axios

  CRON JOB SKELETON:
    cron.schedule('0 */30 * * * *', async () => {
      // Redis lock: prevent overlapping runs
      const lock = await redis.set('trending_job_lock','1','NX','EX',300);
      if (!lock) return;
      try {
        const articles = await fetchFromNewsAPI();         // NewsAPI call
        const factChecks = await fetchFromGoogleFactCheck(); // FC API call
        const all = [...articles, ...factChecks];
        for (const item of all) {
          const result = await axios.post(
            'https://eyesee11-b-ware.hf.space/verify',
            { text: item.claimText }
          );
          if (['false','misleading'].includes(result.data.verdict)) {
            await TrendingStory.findOneAndUpdate(
              { sourceUrl: item.url },   // dedup key
              { headline, verdict, confidence, dangerScore, ... },
              { upsert: true }
            );
          }
        }
        await redis.del('trending_feed');
      } finally {
        await redis.del('trending_job_lock');
      }
    });
