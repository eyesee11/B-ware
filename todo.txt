PART 3b: TRENDING RUMOURS FEATURE — FULL SPEC
================================================================================

PURPOSE:
  A public-facing page that continuously monitors news feeds and fact-check
  databases, auto-runs the RAV engine on every story, and surface the
  most DANGEROUS / MISLEADING stories right now — ranked by harm potential.

DANGER SCORE FORMULA:
  danger_score (0–100) = weighted sum of:
    verdict_weight   : accurate=0, misleading=40, false=80, unverifiable=15
    confidence_bonus : confidence * 20  (max 20 points)
    recency_bonus    : if published < 2h ago → +10, < 24h → +5, else 0
    spread_bonus     : (source_count - 1) * 5  (penalise widely-spread falsehoods)

  e.g. a false claim (80) with 0.9 confidence (+18) from 3 sources (+10)
       published 1h ago (+10) → danger_score = 118 → cap at 100

MYSQL TABLE — trending_stories:
  CREATE TABLE trending_stories (
    id              INT AUTO_INCREMENT PRIMARY KEY,
    headline        TEXT NOT NULL,
    claim_text      TEXT,                  -- extracted claim from headline
    source_name     VARCHAR(255),          -- "Times of India", "NDTV", etc.
    source_url      TEXT,
    published_at    DATETIME,
    fetched_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    verdict         ENUM('accurate','misleading','false','unverifiable'),
    confidence      FLOAT,
    danger_score    FLOAT DEFAULT 0,
    metric          VARCHAR(100),
    official_value  FLOAT,
    claimed_value   FLOAT,
    pct_error       FLOAT,
    explanation     TEXT,
    evidence_json   JSON,                  -- full evidence array from NLP service
    tier_used       VARCHAR(20),
    is_active       TINYINT(1) DEFAULT 1,  -- soft delete
    INDEX idx_danger (danger_score DESC),
    INDEX idx_fetched (fetched_at DESC)
  );

NEWS SOURCES TO PULL FROM (backend fetches these, not frontend):
  Primary:
    NewsAPI: q='India economy OR GDP OR inflation' + country=in
    Google Fact Check: query='India' → latest fact checks from AltNews,
                       FactChecker.in, Boom, AFP India, NDTV Fact Check
  Secondary (add later):
    GNews API (free, 100/day): https://gnews.io
    The Hindu RSS feed: https://www.thehindu.com/feeder/default.rss
    PIB (Press Information Bureau): https://pib.gov.in/allRel.aspx
    Moneycontrol API / RSS

BACKEND FLOW:
  Every 30 min (node-cron job):
    1. GET NewsAPI ?q=India economy&pageSize=20
    2. GET Google Fact Check ?query=India&pageSize=20
    3. Deduplicate by headline similarity (skip if URL already in DB)
    4. For each article:
         a. POST localhost:5001/analyze  → extract claim sentences
         b. Take highest-confidence claim
         c. POST localhost:5001/verify   → get verdict + evidence
         d. Compute danger_score
         e. INSERT into trending_stories
    5. DEL Redis key: trending_feed  (force cache refresh)
    6. Mark stories older than 48h: SET is_active=0

REST API ENDPOINTS:
  GET /api/trending
    → Returns top 50 active stories, ordered by danger_score DESC
    → Cached in Redis (trending_feed) with 5-minute TTL
    → Response: { stories: [...], last_updated, total }
    → PUBLIC endpoint (no auth required — home page widget)

  GET /api/trending/:id
    → Full detail for one story including evidence array
    → No Redis cache needed (rarely called)

  POST /api/trending/refresh
    → Admin only — trigger manual refresh now
    → Useful for testing without waiting for cron

FRONTEND PAGE — /trending:
  Layout: Card grid, sorted by danger_score
  Each card shows:
    - Headline (truncated to 2 lines)
    - Source name + published time ("2h ago")
    - Verdict badge (color-coded: red=false, orange=misleading, green=accurate)
    - Danger score bar (0–100, red gradient)
    - Claimed vs Official value comparison (if numeric)
    - "View full analysis" → expands evidence accordion
  Filter chips at top: ALL | FALSE | MISLEADING | ACCURATE | UNVERIFIABLE
  Sort toggle: DANGER SCORE | RECENCY
  Auto-refreshes every 5 minutes (poll GET /api/trending)

  INSTALL:
    npm install ioredis rate-limit-redis express-rate-limit
    (ioredis is the modern Redis client for Node.js — faster than redis package)

  REDIS VS MYSQL DECISION RULE:
    "Do I need to query this data with SQL joins or filtering?"  → MySQL
    "Do I need this data in under 1ms with no join?"             → Redis
    "Is this data derived/computed and expensive to recompute?"  → Redis cache
    "Does this data need to survive a Redis restart?"            → MySQL
    Redis is a CACHE LAYER on top of MySQL, not a replacement.
    Always write to MySQL first, then cache in Redis.
