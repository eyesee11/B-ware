PRIORITY 2 — ROBUSTNESS & PRODUCTION READINESS
================================================================================

  [x] N-13 INPUT SANITIZATION / CLAIM PREPROCESSING
           File: nlp-service/extractor.py (new function: preprocess_claim)
           Why:  User-submitted text can contain HTML tags, excessive whitespace,
                 zero-width Unicode characters, or emoji that break regex patterns.
           What to do:
             - Strip HTML tags (regex or html.parser)
             - Normalize whitespace (collapse multiple spaces, strip \t \r)
             - Remove zero-width chars (\u200b, \u200c, \ufeff)
             - Normalize Unicode (NFC normalization via unicodedata.normalize)
             - Call this function at the TOP of extract_all() and in main.py endpoints

  [x] N-14 PIPELINE TIMEOUT GUARD
           File: nlp-service/verifier/verdict_router.py
           Why:  A full Tier 1→2→3 pipeline can take 15-25 seconds if all APIs
                 are slow. There's no global timeout — the user waits forever.
           How:
             Use asyncio.wait_for() around route_verification with a 30s cap:
               result = await asyncio.wait_for(
                   route_verification(text, force_tier3),
                   timeout=30.0
               )
             On timeout, return verdict="unverifiable" with explanation
             "Verification timed out after 30 seconds."

  [x] N-15 ENHANCED HEALTH CHECK
           File: nlp-service/main.py (/health endpoint)
           Why:  Current /health just returns {"status": "healthy"} always.
                 Backend needs to know if the service is ACTUALLY ready.
           What to check:
             - Is BART model loaded? (check if _load_pipeline cache is warm)
             - Are API keys configured? (GEMINI_API_KEY, NEWS_API_KEY set?)
             - Return component statuses:
               {"status": "healthy", "bart_model": "loaded",
                "gemini_key": "configured", "newsapi_key": "missing"}

  [x] N-16 STRUCTURED LOGGING (replace print statements)
           Files: All files in nlp-service/
           Why:  print() is fine for development, but in production:
                 - No log levels (can't filter INFO vs ERROR)
                 - No timestamps
                 - Can't route to files or monitoring tools
           How:
             import logging
             logger = logging.getLogger("bware.nlp")
             logger.info("Model loaded")  instead of  print("[Tier2] Model loaded")
           Places with print() that need replacing:
             - tier2_nli.py: _load_pipeline() has 2 print statements
             - main.py: __main__ block has 2 print statements

  [x] N-17 RESULT CACHING AT VERIFICATION LEVEL
           File: nlp-service/verifier/verdict_router.py
           Why:  Same claim verified twice = duplicate World Bank + NewsAPI +
                 Gemini calls. evidence_fetcher has its own cache, but full
                 verification results aren't cached.
           How:
             Add a _TtlCache (like in evidence_fetcher) keyed on (text_hash, force_tier3).
             TTL: 1 hour. Saves API quota and reduces response time.
           Note: Backend will also cache in Redis. This is a local L1 cache
             to avoid hitting the NLP service's own external APIs redundantly.

  [x] N-18 RATE LIMITING ON NLP SERVICE
           File: nlp-service/main.py
           Why:  /verify/deep triggers Gemini (rate-limited to 15 req/min).
                 Without rate limiting, a burst of requests can exhaust the quota.
           How:
             pip install slowapi
             from slowapi import Limiter
             from slowapi.util import get_remote_address
             limiter = Limiter(key_func=get_remote_address)
             @app.post("/verify/deep")
             @limiter.limit("10/minute")
             async def verify_deep(request, response): ...
           Note: Backend also rate-limits, so this is defense-in-depth.

PRIORITY 3 — EXTRACTION QUALITY IMPROVEMENTS
================================================================================

  [x] N-1  FISCAL YEAR SUPPORT in extract_year()
           File: nlp-service/extractor.py
           e.g. "FY2024-25" → 2025, "2023-24" → 2024
           Regex: FY\s?\d{4}(-\d{2})? → resolve to ending year

  [x] N-2  WORD-FORM NUMBER SUPPORT in extract_value()
           File: nlp-service/extractor.py
           e.g. "1.4 billion" → 1400000000.0, "₹2 lakh crore" → 2e12
           Multiplier map: lakh=1e5, crore=1e7, lakh crore=1e12,
                           million=1e6, billion=1e9, trillion=1e12

  [x] N-7  PRE-COMPILE REGEXES using re.compile()
           Files: nlp-service/metrics.py, nlp-service/extractor.py
           Currently recompiled on every call — compile once at module load

  [x] N-9  IMPROVE CONFIDENCE SCORING FORMULA
           File: nlp-service/extractor.py
           Current: (fields_found / 3) * metric_confidence — too coarse
           Better: metric 50% + value 30% + year 20% weighting

  [x] N-19 MULTI-COUNTRY SUPPORT in tier1_numeric.py
           File: nlp-service/verifier/tier1_numeric.py
           Why:  Currently hardcoded to DEFAULT_COUNTRY = "IND".
                 Claims like "US GDP grew 3.1%" would still compare against India.
           How:
             - Add country extraction to extractor.py (regex for country names)
             - Map country names → ISO 3166 alpha-3 codes
             - Pass country to tier1_numeric_check()
           Scope: Start with top 10 countries (US, UK, China, Japan, Germany, etc.)

  [x] N-20 PERCENTAGE vs ABSOLUTE VALUE DISAMBIGUATION
           File: nlp-service/extractor.py
           Why:  "Population reached 1.4 billion" → value = 1.4, but that's
                 not a percentage. "GDP grew 7.5%" → value = 7.5, and that IS
                 a percentage. Different metrics need different comparison logic.
           How:  Add a `value_type` field to extraction output: "percentage" | "absolute"
                 Determine based on: presence of % symbol, metric type
                 (population is always absolute, growth rates are always %)
